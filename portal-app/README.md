# LotusMirk

One app, two halves:

- **Public marketing site** (`app/(marketing)`) — Home, Services, About,
  Contact. Motion is GSAP + Lenis, modeled on SpaceX/Northrop-Grumman-style
  cinematic scroll reveals.
- **Client/vendor portal** (`app/portal`) — authenticated, Clerk
  organizations tagged `client` or `vendor` via `publicMetadata.type`. Each
  signed-in user lands on the dashboard matching their active
  organization's type, and can view project **deliverables**: 3D models
  (`.obj`, viewed inline via Three.js), aerial imagery (`.ecw`, downloaded —
  see note below), 360° tour links, and video/photo galleries.

There is no separate Webflow site — an earlier plan used Webflow for the
marketing pages, but Webflow can't be scripted into existence (a site has
to be created by hand in its own UI first) and its Memberships/user-accounts
feature was sunset Jan 29, 2026, so it couldn't have hosted the portal
either way. Everything now lives in this one Next.js codebase instead.

## Stack

- **Next.js 15** (App Router, TypeScript)
- **Clerk** — authentication, organizations, roles (`@clerk/nextjs`)
- **Supabase** — Postgres database + Storage, via Clerk as a native
  Third-Party Auth provider (no separate Supabase login, no manual JWT
  signing) for the portal; a plain anonymous client for the public contact
  form (see `lib/supabase/public.ts`)
- **GSAP 3.15 + Lenis** — scroll-tied motion on the marketing site only
  (`components/motion/`)
- **Three.js** — inline `.obj` model viewer in the portal
  (`components/portal/ObjViewer.tsx`)
- **Tailwind CSS v4**

## A note on how this was built

This scaffold was written by hand, file by file, rather than generated by
`create-next-app` or installed via `npm install` — this sandbox's npm
registry access returned `403 Forbidden` for `registry.npmjs.org` and `401`
for the only other registry visible in the npm cache, so no package could
actually be downloaded here. That means: the code below is complete and
internally consistent (checked with a bare `tsc` syntax pass — no dependency
types available, but no syntax errors either), but **`npm install`, `next
build`, and `next dev` have not been run or verified in this environment.**
Run `npm install && npm run dev` on your machine, or let Vercel/Railway do it
on deploy, and fix anything that surfaces from an actual dependency
resolution — treat that first install as the real first test of this code.

## Setup, in order

1. **Clerk**
   - Create an application at [dashboard.clerk.com](https://dashboard.clerk.com).
   - Under **Organizations → Settings**, enable Organizations.
   - Copy the publishable key and secret key into `.env.local` (copy
     `.env.local.example` first).

2. **Supabase**
   - Create a project at [supabase.com](https://supabase.com).
   - Copy the project URL and publishable key into `.env.local`.
   - Visit [Clerk's Connect with Supabase page](https://dashboard.clerk.com/setup/supabase)
     to configure the Clerk instance for Supabase compatibility.
   - In the Supabase dashboard, go to **Authentication → Third-Party Auth**
     and add Clerk as a provider.
   - Apply the migrations in order — `0001_init.sql`, `0002_deliverables.sql`,
     `0003_contact_submissions.sql` — via the Supabase connector's migration
     tool, the CLI, or the SQL editor.
   - `0002_deliverables.sql` also creates a private Storage bucket named
     `deliverables`. Upload files under a path of
     `${organization_id}/${project_id}/filename` — that convention is what
     the storage RLS policies match against, so anything uploaded outside
     it won't be readable by the portal.

3. **Install and run**
   ```bash
   npm install
   npm run dev
   ```

4. **Create your first organizations**
   - In the Clerk dashboard (or via `clerkClient().organizations.createOrganization`
     in an onboarding flow you add later), create at least one organization
     with `publicMetadata: { "type": "client" }` and one with
     `publicMetadata: { "type": "vendor" }`.
   - Invite the relevant users to each.

## How the client/vendor split works

- `middleware.ts` only enforces "signed in, with an active organization" —
  it redirects to `/select-org` otherwise. It deliberately does *not* branch
  on org type, because that value lives in Clerk's `publicMetadata`, which
  isn't in the default session token, and middleware runs on the edge
  runtime where an extra API call per request is worth avoiding.
- `lib/org-type.ts` does that lookup once, server-side, via `clerkClient()`.
  `app/portal/page.tsx`, `app/portal/client/page.tsx`, and
  `app/portal/vendor/page.tsx` each call it and redirect away if the active
  org's type doesn't match — so a vendor org can't reach `/portal/client`
  by typing the URL, and vice versa.
- If this needs to scale past a handful of orgs, promote `type` into a
  [customized Clerk session token](https://clerk.com/docs/backend-requests/custom-session-token)
  claim and read it straight from `auth()` instead of calling `clerkClient()`
  on every page load.

## What's still a placeholder

- **Every word of marketing copy** — headline, service descriptions, about
  page, stats — is a stand-in, clearly marked `PLACEHOLDER COPY` in each
  file (`app/(marketing)/**`). Replace with LotusMirk's real services,
  certifications, equipment, and a completed-project case study before this
  goes live. Nothing here was fabricated as fact — it's intentionally
  generic until real content is supplied.
- **All hero/background media is a CSS gradient**, not real footage — swap
  in actual aerial photo/video assets in `app/(marketing)/page.tsx`.
- The `app/portal/client` and `app/portal/vendor` dashboards still have two
  placeholder cards each ("Active projects"/"Invoices",
  "Assigned jobs"/"Payouts") — **Deliverables is wired to real data**
  (`lib/deliverables.ts` + `supabase/migrations/0002_deliverables.sql`), the
  other two are not yet.
- No admin/upload UI for deliverables yet — rows and files currently have to
  be added directly (Supabase dashboard, SQL editor, or Storage upload)
  following the `${organization_id}/${project_id}/filename` path
  convention. Worth building a small internal upload form once this is used
  for real.
- `.ecw` files are download-only by design — there's no practical
  in-browser decoder for that format. If a visual preview matters, generate
  a JPEG/WebP thumbnail alongside the `.ecw` (e.g. via GDAL) and set
  `thumbnail_url` on that deliverable row.
- No billing/Stripe wiring yet. Clerk's B2B bundle includes a full
  Stripe-via-organizations pattern if/when that's needed.
- The public contact form's `contact_submissions` table has no read UI —
  check submissions from the Supabase dashboard/SQL editor for now.

## Deploying

Vercel is the most direct fit for a Next.js app — this project has a
connected Vercel team (`lotusmirk`) ready to deploy to. Railway or Netlify
both work too if you'd rather standardize on the same host as other
services. Set the same environment variables from `.env.local` in
whichever host you pick.
